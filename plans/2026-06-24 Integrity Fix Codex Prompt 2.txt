When a v2 AQI-only repair rebuilds day-partition AQI data under:

```text

```

Required semantics:

1. Connector/day/pollutant AQI data rebuilt

If AQI rows are written or replaced for a connector/day/pollutant, the corresponding AQI timeseries index entries must be refreshed.

2. Connector-scoped AQI repair

If the run is connector-scoped, only the affected connector’s AQI index entries should be rebuilt or updated. Do not require full-day all-connector rebuilds just to update the index.

3. Timeseries index completeness

For an affected timeseries, after repair, the index must point to all relevant AQI rows for the requested range. The website must not continue to report:

```text
required_timeseries_index_miss
timeseries_index_warning
response_complete=false
```

when the underlying v2 AQI day data exists.

4. Preserve other data

Index rebuild/update must not delete unrelated connector/day/pollutant/timeseries entries. If the index is sharded, update only the affected shard(s), or rebuild the shard(s) from authoritative R2 day data.

5. Idempotence

Running the same AQI repair twice should leave the AQI data and timeseries index consistent, without duplicate rows or stale pointers.

6. Post-rebuild validation

The integrity runner should validate not only that the v2 AQI day partition exists, but also that the AQI timeseries index used by `/api/aq/aqi-history` can serve the rebuilt data. If current validation only checks day manifests/data, extend it so AQI index freshness is also checked after an AQI rebuild.
